Occlusion Culling — Portal + Room System

The most important use case is dense ship interiors and space station corridors. A portal-based approach culls entire rooms behind closed walls in O(rooms) per frame — no GPU queries needed.

Top-down view — what the PortalCullingSystem sees

How PortalCullingSystem works each frame

1. Locate cameraPoint-in-AABB test across all rooms → camera's current room (fast, rooms are small)
2. BFS from cameraQueue camera room. Dequeue room → test each portal: is it in frustum AND facing camera? If yes, enqueue adjacent room + clip frustum to portal rect
3. Collect visibleAll rooms reached during BFS go into a visible-room set
4. Cull propsOnly submit entities whose RoomMemberComponent.roomId is in the visible set to ModelBatch

Integration with existing systems

  • ShipInteriorGenerator already builds rooms — it also records doorframe positions as Portal objects and stores the RoomGraph in ShipInteriorComponent
  • Props placed by PropPlacementSystem get a RoomMemberComponent at placement time (which room they're in)
  • When a door opens/closes (event), its portal is flagged open/blocked — a closed door blocks BFS traversal into the next room
  • For exterior scenes (planets, space), this system is inactive — libGDX's built-in frustum culling via ModelBatch is sufficient
  • Characters moving between rooms trigger a room-membership update when they cross a portal rect

Exterior — frustum culling only

  • Outdoors: no occlusion system — ModelBatch already frustum-culls per-renderable
  • Foliage tiles (from Section 3) cull by tile AABB vs frustum — covers large outdoor areas
  • The StreamingSystem (Section 1) handles distance-based load/unload — effectively culls anything beyond the render radius before it's even in memory
Why not GPU occlusion queries? They introduce a 1-2 frame read-back latency, require careful management of conditional rendering, and add significant driver overhead. For compact ship interiors — where a closed wall should cull an entire room's worth of props instantly — portal traversal is O(visible rooms) and zero-latency. It also integrates naturally with the door-open/close event system already in the codebase.
